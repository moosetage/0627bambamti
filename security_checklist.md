# 🔒 바이브코딩 서비스 보안 자가 점검 가이드

본 문서는 AI(바이브코딩 등)로 제작한 학급 서비스, 독서기록장, 퀴즈 앱, 신청폼, 대시보드, 학생 포트폴리오 등의 서비스를 배포하기 전에 직접 점검해볼 수 있는 실무 가이드라인입니다.

> [!WARNING]
> **본인 소유** 또는 **운영자에게 승인받은 서비스**에 대해서만 사용해 주세요. 인가되지 않은 외부 시스템에 대한 무단 스캔/점검은 법적인 책임을 질 수 있습니다.

## 💻 2단계: 터미널 명령어로 직접 점검

아래 명령어의 `your-site.example` 부분을 실제 서비스 도메인(또는 `localhost:포트`)으로 변경하여 터미널에 붙여넣어 실행하세요.

### ① HTTPS 강제 적용 여부
```bash
curl -I http://your-site.example
```
* **정상**: `301 Moved Permanently` 또는 `302 Found`로 `https://` 주소로 리다이렉트됨.
* **위험**: HTTP 상태에서 그대로 웹 대시보드가 열림.

### ② 보안 헤더 적용 여부
```bash
curl -I https://your-site.example
```
* **정상**: 응답 헤더에 `Content-Security-Policy`, `Strict-Transport-Security` (HSTS), `X-Content-Type-Options`가 포함됨.
* **위험**: 주요 보안 헤더 누락 (XSS 및 세션 탈취 취약성이 높아짐).

### ③ 로그인 없는 관리자 페이지 접근 권한
```bash
curl -i https://your-site.example/admin
curl -i https://your-site.example/dashboard
```
* **정상**: `302`, `401 Unauthorized` 또는 `403 Forbidden`.
* **위험**: `200 OK`와 함께 관리자 화면 또는 학생 목록 템플릿이 로드됨.

### ④ 비인증 API 데이터 요청
```bash
curl -i https://your-site.example/api/students
curl -i https://your-site.example/api/dashboard
```
* **정상**: `401 Unauthorized` 또는 `403 Forbidden` 반환.
* **위험**: 학생 정보가 들어 있는 JSON 데이터가 `200 OK`로 즉시 출력됨.

### ⑤ 프론트엔드 코드 내 중요 키 노출 검사
```bash
curl -s https://your-site.example/assets/app.js | rg -n "apiKey|secret|token|password|firebase|supabase|serviceRole|admin"
```
* **정상**: 소스코드에 보안 키가 직접 기재되어 있지 않음.
* **위험**: `serviceRole` 키, supabase `anon` 키가 아닌 `service_role` 중요 토큰, DB 접속 패스워드 등이 검출됨.

### ⑥ 환경변수(.env) 파일 직접 접근
```bash
curl -I https://your-site.example/.env
curl -I https://your-site.example/.env.local
```
* **정상**: `404 Not Found` 또는 `403 Forbidden`.
* **위험**: `200 OK`와 함께 파일 내용(데이터베이스 암호, API 키 등)이 통째로 보임.

### ⑦ 개발 편의 기능 노출 (Swagger / Actuator)
```bash
curl -I https://your-site.example/swagger-ui/index.html
curl -I https://your-site.example/actuator
```
* **정상**: `404 Not Found` 혹은 `403 Forbidden` 처리되거나 접근 제한됨.
* **위험**: API 구조와 세부 라우트 정보가 외부에 노출됨.

## 🤖 3단계: AI 개발 에이전트(Claude Code / Cursor 등)에게 분석 지시하기

터미널 실행이 번거로운 경우, 프로젝트 루트 폴더 내에서 다음 프롬프트를 에이전트에게 전달해 자동으로 취약점을 진단할 수 있습니다.

### 📌 프롬프트 1: 코드 기반 보안 점검 (프로젝트 분석용)
> [!IMPORTANT]
> 에이전트가 검증되지 않은 상태로 코드를 임의 수정하여 장애가 생기지 않도록 방지하는 지시사항이 포함되어 있습니다.

```text
이 프로젝트는 교사가 학생용으로 배포할 웹 서비스야.
아래 항목을 코드에서 점검하고, 발견한 문제를 [위험도 / 파일·위치 / 설명 / 수정 방향] 표로 정리해줘.
중요: 너는 찾아서 보고만 해. 내 확인 없이 코드를 수정하거나 배포·삭제하지 마.
 
1. 로그인 없이 접근 가능한 페이지·API 가 있는지 (서버에서 권한 검사를 하는지)
2. 다른 사용자의 ID 로 바꿔 접근하면 남의 데이터가 보이는 구조인지 (IDOR)
3. API 키·서비스 롤 키·DB 비밀번호가 프론트 코드나 깃 저장소에 들어가 있는지
4. 비밀번호를 평문으로 저장하는지, 인증번호·PIN·로그인 실패에 횟수 제한이 있는지
5. 학생 개인정보(이름·이메일·학급·생년월일)가 공개 페이지·로그·테스트 데이터에 남는지
6. 파일 업로드의 확장자·크기·실행 여부 검증이 있는지
7. 에러 화면에 stack trace·SQL·파일 경로가 노출되는지
```

### 📌 프롬프트 2: 커밋 전 중요 자격 증명(Credentials) 유출 검사
```text
이 저장소의 코드와 git 커밋 기록(.env, 설정파일 포함)에서
API 키, 토큰, 비밀번호, 서비스 롤 키, 인증 정보로 보이는 문자열을 찾아줘.
프론트엔드 번들에 노출되는 값(NEXT_PUBLIC_, VITE_, PUBLIC_ 접두사 포함)은 따로 표시해줘.
찾은 항목만 목록으로 보고하고, 파일을 수정하거나 키를 삭제하지는 마.
```

### 📌 프롬프트 3: 배포 서버 외부 웹 진단 요청
```text
https://your-site.example 는 내가 만든 서비스야.
아래를 curl 로 점검하고 결과(상태코드·헤더 위주)를 정리해줘.
읽기 요청만 하고, 로그인 우회·비밀번호 추측·대량 요청은 하지 마.
 
- http 로 접속 시 https 로 리다이렉트되는지
- 주요 보안 헤더(CSP, HSTS, X-Content-Type-Options)가 있는지
- /admin, /dashboard, /api/students 가 로그인 없이 200 으로 열리는지
- /.env, /.env.local 이 웹에서 열리는지
- /swagger-ui, /actuator 같은 개발용 경로가 노출되는지
 
각 항목을 [정상 / 확인 필요 / 위험] 으로 표시하고, 위험 항목은 이유를 설명해줘.
```

## 📝 4단계: 배포 전 최종 체크리스트

각 항목을 검토한 후 통과 시 체크(`[x]`) 하세요.

### A. 로그인 및 접근 통제 권한
- [ ] 로그인하지 않은 경우 대시보드 접근 차단
- [ ] 관리자 페이지는 오직 관리자 권한을 가진 계정만 진입 가능
- [ ] 학생 계정으로 로그인 시 타인의 학생 기록 열람 불가
- [ ] 교사 계정으로 로그인 시 본인 담당 반/학급 데이터로 제한
- [ ] URL ID 세그먼트(예: `/student/1` -> `/student/2`) 조작 시 타인 정보 차단
- [ ] 프론트엔드 UI 분기뿐 아니라 백엔드 API 단에서도 권한 재검증 수행

### B. 개인정보 보호
- [ ] 이름, 이메일, 생년월일, 학급 정보 등 최소 정보만 수집 및 저장
- [ ] 비인가 사용자에게 노출되는 공개 화면에는 실명 및 학급 정보 마스킹 처리
- [ ] 개발/테스트 데이터베이스에 실사용 학생의 개인정보 대입 금지
- [ ] 애플리케이션 로그에 비밀번호, 토큰, 학부모/학생 연락처 미기록
- [ ] 회원의 탈퇴/삭제 처리 시 연결된 개인정보 데이터 완벽 소거

### C. 비밀번호 및 인증 메커니즘
- [ ] 비밀번호를 단방향 해시(Salt 적용 필수)로 암호화하여 저장 (평문 저장 금지)
- [ ] 비밀번호 초기화 토큰/링크는 단 1회만 사용 가능하며 15분 이내 만료 설정
- [ ] SMS 및 이메일 2차 인증번호의 유효 시간을 최대 3~5분 이내로 제한
- [ ] 로그인 실패 시도 5회 초과 시 계정 잠금 또는 지연 대기 적용
- [ ] 세션 쿠키 설정에 `HttpOnly`, `Secure`, `SameSite=Lax/Strict` 속성 강제 지정

### D. API 및 DB 보안
- [ ] API 접근 시 발급받은 토큰의 유효 기간을 짧게 설정 (Access Token 만료 시간 최소화)
- [ ] 클라이언트가 전달한 ID 값을 그대로 신뢰하지 않고 서버 세션 정보로 대조
- [ ] 모든 데이터베이스 쿼리에 `WHERE tenant_id = ?` 또는 `WHERE user_id = ?` 등의 오너십 조건 적용
- [ ] 일반 권한의 API 토큰으로 어드민 API endpoint 접근 통제

### E. 파일 업로드 검증
- [ ] 업로드 파일 검증 시 확장자뿐 아니라 실제 파일 매직 넘버(MIME Type)까지 검증
- [ ] 1회당 업로드 크기를 최대 제한(예: 10MB)하여 서버 리소스 고갈 방지
- [ ] 업로드된 파일이 서버 내에서 임의 실행(`exe`, `sh`, `php`, `jsp` 등)되지 않도록 실행 권한 제한
- [ ] 업로드된 파일명을 식별하기 어려운 난수값으로 자동 변경 및 저장

### F. 외부 API 키 관리
- [ ] OpenAI, AWS S3, Firebase 등 제3자 API key 값을 프론트 JS 소스에 하드코딩 금지
- [ ] 강력한 접근 권한(Service Role, Admin 권한)을 가진 토큰은 서버 인프라 변수(`process.env`)에만 보관
- [ ] 만약 퍼블릭 리포지토리에 키가 노출된 경우, 즉시 키를 재생성(Rotate) 및 비활성화 처리
- [ ] 무료 및 유료 외부 API 계정 정보에 일일/월간 크레딧 사용 한계값(Quotas) 설정

### G. CORS 및 예외 처리
- [ ] CORS 헤더의 `Access-Control-Allow-Origin`에 와일드카드(`*`)와 `credentials: true`를 동시 설정하지 않음
- [ ] 오류(Exception) 발생 시 스택 트레이스(Stack Trace)나 SQL 쿼리를 화면에 그대로 보여주지 않고 공통 에러 페이지 처리
- [ ] 상세 에러 이력은 시스템 내부 로그 저장소에만 남김

## 🚨 취약점 발견 시 대응 프로토콜

### 🔴 즉각 조치 사항 (심각)
> **대상**: 개인정보 API 노출, `.env` 파일 유출, 관리자 권한 완전 우회 등

1. **서비스 중단**: 즉시 서버 서비스를 내리거나 방화벽으로 차단 페이지로 유도합니다.
2. **토큰 폐기 및 재발급**: 노출된 DB 암호, 외부 API 키 등을 완전히 무효화한 뒤 새롭게 생성하여 서버 환경변수에 적용합니다.
3. **영향도 평가**: 접속 로그(Web log)를 검토하여 노출 기간 동안 외부 침입이나 불법 다운로드 정황이 있는지 로그 IP를 분석합니다.
4. **보고**: 유출 건이 중대할 경우 학교/기관 정보보호 책임자에게 보고하고 후속 조치를 취합니다.

### 🟡 상시 개선 사항 (보통/낮음)
> **대상**: 보안 헤더 부재, 에러 시 500 디버그 정보 노출, 검색 노출 방지 미설정 등

* 다음 정기 배포 또는 코드 푸시 때 패치 내역에 반영하고 배포합니다.
